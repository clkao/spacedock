Make the live-runtime e2e run on PRs without a manual per-PR step. Today `runtime-live-e2e.yml`
(on `next`) gates the live job with `if: workflow_dispatch OR PR-carries-'run-live-e2e'-label`, **on
top of** the CI-E2E* environment required-reviewer approval. So a PR that isn't labeled at creation
never queues the live job at all — the operator has to remember to add the label, or manually
dispatch. Two stacked gates (label + environment) do one job's worth of work.

## The two paths CL named

1. **Mimic old main: always-pending + explicit env approval (recommended).** Drop the `run-live-e2e`
   label condition from the live job's `if`. Every PR to `next` then queues the live job straight
   into `waiting` on its CI-E2E* environment; the maintainer approves the deployment for the PRs
   that should burn budget and ignores/rejects the rest. The environment required-reviewer gate
   already provides the budget safety the label was standing in for, so nothing runs a paid model
   unapproved. The secret-free `offline` job stays the always-run gate. This is exactly the
   "always pending, we explicitly approve env" model.
2. **Auto-label on PR creation (lower blast radius).** Keep the label gate but have the `pr-merge`
   mod apply the `run-live-e2e` label when it opens the PR (`gh pr create --label run-live-e2e`),
   removing the manual step while preserving the current architecture. Falls back to documenting the
   labeling requirement for hand-opened PRs.

## Acceptance criteria

**AC-1 — A PR to `next` queues the live job without any manual labeling step.**
Verified by: open a PR (or simulate the `if` evaluation) and confirm the live job reaches `waiting`
on its environment gate with no label applied — under path 1, by asserting the workflow `if` no
longer references the label; under path 2, by asserting the mod applied the label at PR-open.

**AC-2 — No paid model runs without an explicit approval.**
Verified by: the live job still declares `environment: CI-E2E*` and stays in `waiting` until a
required reviewer approves; assert the environment gate is intact after the change.

**AC-3 — The secret-free offline gate still runs on every PR.**
Verified by: the `offline` job (no environment, no secret) remains unconditional and is the PR's
real pass/fail gate.

## Test plan

Primarily a workflow-YAML change validated by reading the rendered `if`/`environment` conditions and
a dry PR (or `act`/`gh workflow view`); plus, for path 2, a `pr-merge` mod behavioral assertion that
`--label run-live-e2e` is passed. Ideation should pick path 1 vs 2 (FO lean: path 1 — fewer moving
parts, matches the model CL described, env gate already enforces budget safety), and confirm the
fork-PR security model (`pull_request` not `pull_request_target`; same-repo-or-no-secrets) is
unchanged by dropping the label condition.

## Decision (ideation): path 1

**Chosen: path 1 — drop the label clause; rely on the per-variant environment-approval gate.**
This is CL's always-pending-approve-env model, and the empirical footprint analysis below makes it
strictly cheaper and lower-risk than path 2.

### Why path 1 over path 2 (grounded in the actual files)

- **Blast radius is one file nobody else is editing.** The `run-live-e2e` label appears ONLY in
  `.github/workflows/runtime-live-e2e.yml` (and its per-worktree copies) — no mod, no Go code, no
  other workflow references it (`grep -rn run-live-e2e` confirms). Dropping the clause strands no
  other consumer. Every in-flight worktree copy of the file (`audit-host-neutrality-seam`,
  `phaseA-pr`, `claude-runtime-segregation`, `audit-cli-ergonomics`, `k6-pr`) is byte-identical to
  `origin/next` — no concurrent session is mid-edit on it. The two entities the FO flagged as
  possible collisions do NOT edit this file: `split-root-collaboration-remote-resume` (sc) touches
  `dispatch/helpers.go` / `status/handlers.go` / commission only; `roborev-validation-hook` merely
  *rides* `runtime-live-e2e.yml` as a proven pattern (its PR-JOB extension is a separate
  `roborev init gh-action`-generated file). So path 1 is genuinely DISJOINT and lands concurrently
  with both — no serialize-at-implementation needed.
- **Path 2 collides on a hot file.** Path 2 must edit `mods/pr-merge.md:51` (the `gh pr create`
  line), which jx/f2/merge-ceremony work is also touching — a real hot-file collision path 1 avoids.
  Path 2 is also semantically awkward here: `pr-merge.md:51` opens PRs with `--base main`, but the
  live-e2e trigger is `pull_request: branches: [next]`, so auto-labeling the `main`-based per-entity
  PRs would not even target the lineage that runs the live job.

### The exact `runtime-live-e2e.yml` edit (NEXT lineage)

Target baseline = `origin/next:.github/workflows/runtime-live-e2e.yml` (the PR-triggered,
label-gated, two-variant matrix — NOT the `main` `workflow_dispatch`-only single-env file; see
"main-vs-next divergence" below).

1. **Delete the live-job `if:` line.** Today:

   ```yaml
   if: ${{ github.event_name == 'workflow_dispatch' || contains(github.event.pull_request.labels.*.name, 'run-live-e2e') }}
   ```

   Remove it entirely. With no `if`, the `claude-live` matrix runs on BOTH declared triggers
   (`workflow_dispatch` and `pull_request: branches: [next]`). The `on:` block has only those two
   triggers, so no narrower replacement condition is needed — deleting the line is the whole change.

2. **Update the two comments that describe the now-removed label gate so they describe the code as
   it is** (the file's SECURITY MODEL header and the per-job comment above the `if`). Specifically:
   - Top-of-file header: drop the sentence "The live job is additionally label-gated: it runs only
     on a manual dispatch or a PR carrying the `run-live-e2e` label." and rewrite the trailing
     sentence so the gate it names is "same-repo-or-no-secrets + the per-variant environment
     approval" (drop "+ the label").
   - The `claude-live` per-job comment: replace the "runs only on an explicit opt-in … or a PR
     carrying the `run-live-e2e` label … this job adds the label gate ON TOP OF the per-variant
     environment approval" text with prose stating that every PR to `next` (and every manual
     dispatch) queues the live matrix, and each variant pauses in `waiting` on its CI-E2E* /
     CI-E2E-OPUS environment until a maintainer approves THAT deployment.

   No code comment may claim a label gate after the `if` is gone.

  Verified by asserting the rendered live-job `if` no longer references the label (it is absent),
  and `pull_request: branches: [next]` is retained.
- No paid model runs unapproved: each leg still declares `environment: ${{ matrix.environment }}`
  (CI-E2E* required-reviewer gates), intact after the edit. Verified by reading the matrix +
  `environment:` block post-change.
- The `offline` job is unchanged: NO environment, NO secret, runs unconditionally on every PR as the
  real pass/fail gate. Verified by confirming the `offline` job body and its lack of an `if`/`environment`.
- Fork-PR security model unchanged: trigger stays `pull_request` (NOT `pull_request_target`), so
  GitHub withholds repo secrets from fork PRs; same-repo-or-no-secrets is preserved. Dropping the
  label condition does not touch the trigger or `permissions:`. Verified by confirming the `on:`
  block and `permissions: contents: read` are unchanged.
