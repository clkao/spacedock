# Integrate Skills With Launcher

Point first-officer and ensign skill instructions at the `spacedock` launcher (instead of the plugin-private `skills/commission/bin/status` path), and fix the dispatch helper so split-root, folder-form entities are dispatched correctly. This entity is the home of two confirmed FO decisions: (A) the split-root worktree contract amendment, and (C) the slug-not-stem dispatch-derivation fix.

## Problem Statement

Today the first-officer and ensign skills are plugin-shipped (the `spacedock:first-officer` / `spacedock:ensign` skill namespace), and their instruction surface is wired to plugin internals in two ways that are wrong for this project:

**1. Hardcoded plugin-private status path.** The FO reference files address the status viewer as `{spacedock_plugin_dir}/skills/commission/bin/status`. Concretely this appears in `first-officer-shared-core.md` (startup `--discover` at line 8; the Status Viewer section lines 25-55; dispatch `--set` at line 99; merge/mod-block calls) and in `claude-first-officer-runtime.md` (the break-glass and `claude-team build` invocations, e.g. line 95). The dispatch helper `skills/commission/bin/claude-team` also emits and depends on that path. The roadmap (§4) and `AGENTS.md` require the skill surface to call the stable `spacedock` launcher instead, so the project is decoupled from plugin internals and from whichever script version happens to be installed.

**2. Dispatch derivation breaks on this project's split-root, folder-form entities.** This workflow runs in the hardest configuration the helper supports, and two helper behaviors are wrong for it:

- *Split root.* The workflow README lives in the main repo (`docs/dev/README.md`) with `state: .spacedock-state`; runtime entities live in a **separate, nested git repository** at `docs/dev/.spacedock-state/` which is git-ignored from the main repo (verified: `docs/dev/.spacedock-state/.git` exists and `git check-ignore docs/dev/.spacedock-state` matches). A `git worktree` of the *main* repo therefore does not even contain the `.spacedock-state` entity files. The helper's `cmd_build`, when a worktree is present, rewrites the entity path to `{worktree_path}/{entity_rel}` (`claude-team:380-390`) and points the ensign's completion signal at that worktree copy (`claude-team:438`). In split-root that path does not exist; the entity body and stage reports must be written and committed to the shared `.spacedock-state` checkout, with only **code** isolated in the worktree.

- *Folder-form slug.* Entities here are folder-form (`{slug}/index.md`, e.g. `skill-integration-vendor-branch/index.md`). The helper derives `slug = os.path.splitext(os.path.basename(entity_path))[0]` (`claude-team:317`), which yields `index` for every folder-form entity instead of the entity slug. That `index` then flows into the worker name `derived_name = {worker_key}-{slug}-{stage}` (`:318`), the git branch `{worker_key}/{slug}` (`:371`), and the dispatch-file path `/tmp/spacedock-dispatch/{derived_name}.md` (`:458`). The result is that every folder-form entity dispatches to the same `…-index-…` worker name, the same `…/index` branch, and a colliding dispatch file — names that are both ambiguous across entities and meaningless to a human. The status script's own slug rule is the correct one: folder name when the file is `index.md`, filename stem otherwise (`skills/commission/bin/status:436-445`).

The two problems are coupled and co-owned here because the fix is one change to the dispatch contract: route the status path through `spacedock status` AND make the dispatch helper produce correct slug-derived names and a state-checkout entity path under split root. A sibling stage (`vendor-status-compatibility`) is concurrently designing `spacedock status` itself; this stage consumes that command surface and does not build it.

## Proposed Approach

**Mechanism: vendor the skill surface onto a project branch.** The project carries its own copies of the skill instruction files and the `claude-team` dispatch helper under the project's `skills/` tree (the directory `AGENTS.md` already reserves for "skill-facing command integration notes and fixtures"), amended so the instructions call `spacedock status` and the helper derives dispatch correctly. The harness loads the project-vendored skill in preference to the plugin-shipped one. This is "vendor branching": the project owns a branch of the skill contract, version-pinned with the launcher, exactly as `vendor-status-compatibility` vendors the status script rather than delegating to the live plugin path. Vendoring (not editing the installed plugin in place) is what makes the change reviewable, reproducible, and testable with checked-in fixtures.

Three amendments ride the vendor branch:

**A. Status-path text → `spacedock status`.** Every `{spacedock_plugin_dir}/skills/commission/bin/status` occurrence in the vendored FO reference files becomes `spacedock status`, and the vendored `claude-team` helper emits/uses `spacedock status` (including the `--discover` startup call and any break-glass text). Because `vendor-status-compatibility` makes `spacedock status` an argv-transparent pass-through of the current script — same `(argv, env, cwd) -> (stdout, stderr, exit code, fs mutations)` contract for every FO-load-bearing flag (`--workflow-dir`, `--boot`, `--next`, `--next-id`, `--set`, `--archive`, `--resolve`, `--validate`, `--short-id`, `--where`, `--archived`, `--discover`) — this is a textual substitution with no behavioral change. The FO's `--boot` section parsing, `--set` stdout narration, and exit-code gating all keep working unchanged. **Assumption stated for the gate:** `spacedock status <args...>` is a drop-in replacement for `{plugin}/skills/commission/bin/status <args...>` on every flag the FO uses; if `vendor-status-compatibility` lands a narrower surface, this stage's text substitution must be re-scoped to only the supported flags.

**B. Split-root worktree contract amendment (this entity is its home).** Amend the worktree-ownership contract — in the vendored `ensign-shared-core.md` / `first-officer-shared-core.md` Worktree Ownership sections and in the vendored `claude-team` `cmd_build` — to read: *when the workflow is split-root (`state:` is set on the workflow README), worktree stages isolate CODE only. The entity body and stage reports are written and committed to the shared state checkout, never to a worktree copy.* Concretely in `cmd_build`: when split-root, the entity-read instruction and completion-signal `entity_file_ref` point at the state-checkout entity path (the path the FO passed, unchanged), NOT `{worktree_path}/{entity_rel}`. The worktree instructions (working dir, branch, "commits MUST be on this branch") still apply to code. The narrow existing `pr:`-mirrored-on-main exception is unaffected. Non-split-root workflows keep today's behavior (entity lives in the worktree copy).

*Concurrency-safe state commits (part of amendment B).* Because every worktree stage and every non-worktree stage now commits its entity body and stage report to the **one shared, non-branched `.spacedock-state` checkout**, concurrent stages race on a single git index. A bare `git add -A` / `git commit` from one writer sweeps up a sibling writer's already-staged entity file, cross-attributing (and potentially clobbering) another entity's change — observed this session: one ensign's bare commit swept in a sibling's staged file with no data loss but crossed attribution. The vendored ensign/FO contract MUST therefore require concurrency-safe state commits, in preference order:
- *Preferred — tool-managed atomic state commits.* The status tool owns the `add`+`commit` of an entity's state under a lock (it already owns `--set`/`--archive` mutations), serializing concurrent state writes so parallel writers never cross-attribute or clobber. This pairs with `native-go-status`'s decision-B atomic-write principle (the tool owning serialized state writes) — assumption stated for the gate: if `native-go-status` lands a tool-owned state-commit entry point, the vendored skill contract routes entity commits through it.
- *Fallback — path-scoped commits per writer.* Until the tool owns commits, every writer stages and commits ONLY its own entity path: `git -C {state_checkout} add {entity_path} && git -C {state_checkout} commit -- {entity_path} -m "…"`. Never a bare `git add -A` or bare `git commit` in the shared state checkout. Retry on `index.lock` contention after a short wait (~2s).

**C. Slug-not-stem dispatch derivation.** In the vendored `claude-team` `cmd_build`, derive the slug with the status script's canonical rule: if `basename(entity_path) == "index.md"`, `slug = basename(dirname(entity_path))`; else `slug = splitext(basename(entity_path))[0]`. The corrected slug feeds `derived_name`, the branch `{worker_key}/{slug}`, and the dispatch-file path, so folder-form entities get human-meaningful, per-entity-unique worker names, branches, and dispatch files. The entity path the FO passes is already the state-checkout path; the helper resolves the ensign's entity reference to that state-checkout path (per amendment B) rather than a worktree copy.

**Boundaries (compatibility-first).** No PR merge mod, lifecycle mod, or mod-hook behavior is introduced (out of scope per roadmap and `AGENTS.md`). Native `state:` resolution inside `spacedock status` is Stage 6, not here — during this stage the compatibility symlink (`.spacedock-state/README.md -> ../README.md`) still lets `spacedock status --workflow-dir docs/dev/.spacedock-state` operate, so the launcher command and the skill substitution are independent of split-root status resolution. Amendments B and C are about the **dispatch helper** (how ensigns are launched and where they write), which is independent of how `spacedock status` resolves roots.

## Acceptance Criteria

Each AC names a property of the finished entity and how it is verified.

**AC-1 - No vendored skill instruction text references the plugin-private status path.**
After this stage, no file in the project's vendored skill surface (the vendored FO/ensign reference files and the vendored `claude-team` helper) contains the substring `skills/commission/bin/status`; every status invocation in that text reads `spacedock status`.
Verified by: a static skill test that greps the vendored skill tree for `skills/commission/bin/status` and asserts zero matches, and greps for `spacedock status` and asserts the FO status invocations are present (regression check named in roadmap §4 and the Test Gates).

**AC-2 - First-officer instructions invoke the launcher for status, with flags preserved.**
The vendored FO reference text issues status reads and mutations as `spacedock status --workflow-dir {workflow_dir} [flags]` for the FO-load-bearing flag set (`--discover`, `--boot`, `--next`, `--next-id`, `--set`, `--archive`, `--resolve`, `--validate`, `--short-id`, `--where`, `--archived`), preserving each flag's role.
Verified by: static skill tests asserting the launcher-prefixed invocation appears for startup discovery, `--boot`, dispatch `--set`, and merge `--archive`; plus the smoke test in AC-5 exercising the live flags end-to-end.

**AC-3 - Ensign dispatch hands the `.spacedock-state` entity path, even under a worktree stage.**
For a split-root workflow, the dispatch assembled by the vendored `claude-team build` instructs the ensign to read and write the entity at its `.spacedock-state` path, and the completion signal references that same state-checkout path — never a `{worktree}/…/index.md` copy. Worktree code-isolation instructions (working dir, branch) are still emitted for `worktree: true` stages.
Verified by: a `claude-team build` fixture test that, given a split-root folder-form entity and a worktree stage, asserts the emitted dispatch body's entity-read line and completion-signal `entity_file_ref` equal the state-checkout path and contain no `.worktrees/` segment, while the worktree working-dir/branch lines are still present.

**AC-4 - Dispatch worker name, branch, and dispatch-file are derived from the entity slug, not the filename stem.**
For a folder-form entity `{slug}/index.md`, the worker name is `{worker_key}-{slug}-{stage}`, the branch is `{worker_key}/{slug}`, and the dispatch file is `/tmp/spacedock-dispatch/{worker_key}-{slug}-{stage}.md` — all using the folder `{slug}`, not `index`. Flat `{slug}.md` entities continue to derive `{slug}` from the filename stem.
Verified by: a `claude-team build` derivation test asserting, for a folder-form fixture entity, that the emitted `name`, the branch line in the dispatch body, and the `dispatch_file_path` all contain the folder slug and never the literal `index`; and a flat-entity case asserting the stem-derived slug is unchanged.

**AC-5 - A pilot split-root workflow can list, mutate, and archive an entity through the launcher.**
Against a pilot workflow in `state:` split-root layout (the symlink compatibility profile is sufficient for this stage), `spacedock status` renders the entity table, `spacedock status --set {slug} field=value` mutates entity frontmatter in the state checkout, and `spacedock status --archive {slug}` moves the entity under `_archive` in the state checkout — each producing the current script's output and exit code.
Verified by: a smoke test (fixture-backed, copying a folder-form-entity workflow into a temp state checkout) that runs list, `--set`, and `--archive` through the launcher and asserts the entity table row, the `field: old -> new` narration, and the archive move land in the state checkout.

**AC-6 - No PR-merge, lifecycle-mod, or mod-hook behavior is introduced.**
The change set touches only skill instruction text and dispatch derivation; it adds no `## Hook:` mod, no PR merge flow, and no lifecycle-hook handling.
Verified by: review check plus a grep asserting the diff introduces no new `## Hook:` headings or PR-merge invocations in the vendored skill surface.

**AC-7 - State commits to the shared `.spacedock-state` checkout are concurrency-safe — no bare-commit cross-attribution between concurrent writers.**
The vendored ensign/FO contract requires that entity-body and stage-report commits to the shared, non-branched state checkout never use a bare `git add -A` / `git commit`: either the status tool owns the add+commit under a lock (preferred), or each writer uses a path-scoped commit (`git -C {state_checkout} commit -- {entity_path}`). Two ensigns committing different entities concurrently cannot cross-attribute or clobber each other's staged files.
Verified by: a static skill test asserting the vendored ensign/FO commit instructions specify path-scoped or tool-owned commits and forbid bare `git add -A`/`git commit` in the state checkout; and a concurrency regression test that stages entity A, then runs the prescribed commit for entity B in the same checkout, and asserts entity A's staged change is NOT swept into entity B's commit.

## Test Plan

Proof sits at the level of each claim. The skill-text and dispatch-derivation claims are about **instruction text and helper output**, so they are static skill tests and `claude-team build` fixture tests, not Go tests. The launcher-behavior claim (AC-5) is a fixture-backed smoke test through `spacedock status`. The baseline `go test ./...` gate (`AGENTS.md`) still runs for any launcher code touched, but this stage is expected to add no Go product code — only skill text, the vendored helper, and tests/fixtures. Estimated complexity: moderate; the risk is concentrated in the dispatch-derivation fixtures (folder-form + split-root + worktree), which is exactly where the validate-the-mechanism-first discipline applies.

- **Static skill text tests (AC-1, AC-2, AC-6).** Grep the vendored skill tree: assert zero `skills/commission/bin/status` matches; assert `spacedock status` appears for startup `--discover`, `--boot`, dispatch `--set`, and merge `--archive`; assert no new `## Hook:` heading or PR-merge invocation. Cheap; runs in milliseconds; pins the exact before/after wording the roadmap regression check requires.
- **Dispatch-derivation fixture tests (AC-3, AC-4) — validate first.** Drive the vendored `claude-team build` with a checked-in folder-form entity in a split-root fixture and a `worktree: true` stage, then assert on the emitted JSON + dispatch body: (a) `name`, branch line, and `dispatch_file_path` use the folder slug not `index`; (b) the entity-read line and completion-signal ref point at the state-checkout path with no `.worktrees/` segment; (c) worktree working-dir/branch lines are still emitted. Add a flat-entity case to lock the stem path. This is the riskiest contract (split-root + folder-form + worktree all at once) and is the smallest end-to-end exercise of the changed dispatch path — run it before the broader smoke test.
- **Launcher smoke test (AC-5).** Copy a folder-form-entity workflow into a temp state checkout (symlink profile), then run `spacedock status` (list), `spacedock status --set {slug} field=value`, and `spacedock status --archive {slug}`; assert the table row, `field: old -> new` narration, and the archive move into `_archive` in the state checkout. Uses the real launcher and the real (vendored) status behavior — no mocks — proving the substituted command actually drives a list/mutate/archive cycle.
- **Concurrency-safe-commit tests (AC-7).** A static skill test asserts the vendored ensign/FO commit instructions are path-scoped or tool-owned and forbid bare `git add -A`/`git commit` in the state checkout. A regression test in a temp git checkout stages entity A's file, then performs the prescribed commit for entity B (path-scoped `git commit -- {B}` or the tool-owned path), and asserts A's staged change is still staged afterward — i.e. it was not swept into B's commit. Cheap; reproduces the observed cross-attribution hazard directly so a regression to bare-commit is caught.

This stage does not test native `state:` split-root status resolution (Stage 6) or native Go status parity (Stage 5); it relies on the compatibility symlink for status resolution and on `vendor-status-compatibility` for the `spacedock status` surface.
