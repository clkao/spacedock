## Problem

Encode the hard-won deliverable principles into the SHIPPED spacedock contract so future dev work cannot drift into the doc-only / grep-proof antipatterns, and so the discipline travels with the published plugin rather than depending on any one user's environment. Design inputs (both in `docs/dev/_proposals/`): `encoding-deliverable-principles.md` (the four principles + the code guard, with exact before/after wording) and `tdd-and-template-adoption.md` (template comparison + ergonomics) — **with that study's "rely on global CLAUDE.md" TDD conclusion SUPERSEDED** (see AC-3 / Notes).

The four principles: (P1) no doc-only deliverable — every AC has an oracle external to itself; (P2) proof is behavioral, not grep; (P3) enforce in code, not prose; (P4) spike the riskiest unknown in ideation. Plus a generalizing meta-principle surfaced this sprint: **PORTABILITY / self-sufficiency** — the shipped contract assumes nothing from the user's environment (no per-user CLAUDE.md, no Python-on-PATH in the dispatch path, no plugin-private paths). The TDD correction is an instance: TDD authoring discipline must live in the portable `spacedock:ensign` contract, not global CLAUDE.md.

Plus FO-POSTURE principles (captain, "What Awesome Looks Like") — distinct from the deliverable principles (those govern entity proof; these govern how the FO operates), and equally portable (shipped FO contract, not CLAUDE.md): (1) **begin with the end** — name the observable end value before the work, the orchestration analog of P1; (2) **bring a yes-able ask** — lead with the decision + recommendation, supply rationale + evidence to vote yes, cut the rest (generalizes the existing Gate Presentation discipline to every approval/escalation); (3) **JFDI, no doc ceremony** — take the obvious in-authority action without asking; produce no document that isn't a deliverable-with-an-oracle or analysis feeding a decision. #2 and #3 compose, not conflict: JFDI the reversible/in-authority work, reserve the captain for genuine forks *with* a yes-able proposal (= the handoff's auto-approve-vs-escalate rule).

## Proposed approach

**Doc-only ban (P1, operationalized — concise form for the contract).** Every entity ships a behavior change — code, test, config, a captured measurement, or a decision the captain records; a design alone is the IDEATION output of an implement-bound entity, never its own terminal stage. Captain veto before code lands is the IDEATION GATE itself — for a veto-needing entity the FO presents that gate for explicit approval rather than auto-approving — so there is no doc-only ceremony (file → worktree-to-write-markdown → gate-on-prose → "now implement it too" → pay twice). This workflow needs NO new `auto-approve` frontmatter field, `plan` stage, or `research-spike` entity-type (those are another workflow's mechanisms): here the ideation gate is the veto, P4 is the in-ideation spike, and a genuinely code-free decision lives as an ADR/roadmap entry, not a dev-queue entity. (external-tracker-checkpoint is the live cautionary tale: it should have folded into its siblings' ACs + an ADR.)

**Wording (shipped contract): plain language, no jargon.** Do NOT use the term "oracle" in the README, FO contract, or ensign contract — say plainly: proof must be EXTERNAL to the entity and able to FAIL — a test, a command's exit code/output, or resulting on-disk state — not a re-reading of the entity's own prose. The FO contract carries only the GATE CHECK (at each gate, confirm every AC's proof meets that bar; reject ACs proven only by re-reading the entity), in the register of the existing AC-coverage cross-check — not a test-theory lecture; the fuller "what makes a good test" guidance lives README-side (author-facing) and ensign-side (worker-facing). Name the guard plainly (e.g. self-reference check), not "self-oracle lint." (This seed uses shorthand for brevity; the SHIPPED contract text must be plain.)

## Acceptance criteria

Each AC names a property of the finished entity, not a stage action, and how it is verified by an exercise-and-observe check external to this entity body. The load-bearing deliverable is the CODE GUARD (AC-1) — its red/green tests, driving the real binary against fixture entities, are this entity's external behavioral proof, so the entity is NOT itself doc-only. The contract-encoding ACs (AC-3) are honest about their ceiling (wording-present, P3) and lean on AC-1's guard + the behavioral-proof gate-check for teeth. Note: "self-reference check" is the guard's plain name; "self-oracle" is used below only as internal working shorthand for the antipattern (an AC proven by re-reading its own prose) — the SHIPPED contract text uses plain wording (see AC-3), never "oracle".

**AC-1 — A `status --validate` self-reference check plus a terminal-PASSED `--set` guard refuse advancing an entity whose ACs cite no proof external to the entity.**
End state: when an entity's only AC proof is a re-reading of its own prose (the self-reference antipattern), `spacedock status --set <slug> verdict=passed status=done` exits 1, prints the guard error in the existing mod-block idiom, and leaves the entity frontmatter unmutated (still its prior status); `--force` bypasses with the same warning shape as the mod-block `--force`; an entity whose ACs cite external proof reaches `done` cleanly; `spacedock status --validate` flags the self-reference AC with the standard `Error: … workflow= scope= slug= id= path=` evidence line and exits 1, and returns `VALID` for a clean workflow.
Verified by: behavioral tests driving the real binary, modeled on `internal/status/archive_guard_test.go::TestTerminalSetUnderModBlockRejected` and `native_validate_test.go` — launcher-vs-oracle parity on exit code, stderr text, and post-write frontmatter bytes. Three fixtures: (a) self-reference-only entity → `--set verdict=passed` exit 1 + unmutated; (b) external-proof entity → `--set` reaches `done` exit 0; (c) same self-reference entity + `--force` → exit 0 with the override warning. Plus a `--validate` test asserting the evidence line for the self-reference AC and `VALID` for the clean fixture. Load-bearing check: flip the fixture's proof text from self-reference to external and confirm the guard stops firing (the guard keys on the proof clause, not a tautology).

### P4 spike: can the self-reference check be made low-false-positive? — DONE, evidence below

**Riskiest unverified mechanism:** the proposal flags that a regex over `Verified by` prose is itself a grep-over-prose (the very antipattern P2 warns against) and could mis-fire. If the check has poor precision, the terminal-PASSED guard it backs becomes a nuisance the FO routinely `--force`s past, defeating the point. So this is the path that would invalidate AC-1 if it broke — spiked first.

**Exercise (smallest end-to-end):** extracted all **79 real AC items** from the 15 archived + active entities under `.spacedock-state` and ran two candidate classifiers against them, scored against ground truth (the known self-reference case is `external-tracker-checkpoint` AC-6; AC-5 is the grep-over-code case the proposal calls a P2 — not P1 — problem). Reproduction scripts preserved under `deliverable-contract-hardening/artifacts/self-reference-lint-spike/`: `01-naive-regex.py`, `02-refined-classifier.py`, `03-adversarial-cases.py` (run from the `.spacedock-state` root, `python3 deliverable-contract-hardening/artifacts/self-reference-lint-spike/02-refined-classifier.py`).

**Result — the naive regex (proposal's first cut: match `this entity` / `this entity's` / `decision section` anywhere in the AC) has a real false positive:** 3 hits / 79 ACs — (1) external-tracker AC-6 (true positive), (2) external-tracker AC-5 (cites code paths — arguable, a P2 case not a self-reference), and **(3) THIS entity's AC-1, which merely QUOTES the antipattern as the fixture's content** (`"verified by review of this entity's own decision section"`). Hit (3) is the exact mis-fire the proposal feared: a regex firing on a *quoted example* of the antipattern, not a live self-reference.

**Result — the refined classifier is precision 1.0 / recall 1.0 on the real corpus.** Two refinements remove the false positives: (a) **isolate the proof clause** (scan from the `Verified by:`/`Oracle:`/`Proof:` marker onward, not the whole AC), and (b) **strip quoted/backticked spans** before matching (a quoted example of the antipattern is not the live proof), and (c) **require co-absence of an external-proof token** (`test`, `.go`, `exit code`, a `--flag`, `fixture`, `assert`, `parser`, `command/parser code path`, etc.) — an AC that names a self-reference AND a test/command/file is not a pure self-reference. On the 79-AC corpus this flags exactly 1 item: external-tracker AC-6, the true self-reference. Zero false positives. The adversarial edge-case suite (`spike_lint3.py`) confirms all 6 hand-labelled cases classify correctly, including the AC-1 quoted-antipattern (correctly NOT flagged) and the dodge-phrasing false-negative (correctly NOT flagged by the lint — caught instead by the FO cross-check, per the proposal's two-layer composition).

**Conclusion (recorded behavioral evidence):** the self-reference check IS mechanically low-false-positive when it (1) scopes to the proof clause, (2) strips quoted spans, (3) requires external-proof-token absence. The implementation must encode these three refinements, not the naive any-occurrence regex. The known residual false-negative (a self-reference phrased without any literal self-reference token, e.g. "inspection of the recorded rationale") is acceptable and by design: the lint warns / the guard blocks the narrow terminal-PASSED case, and the FO behavioral-proof cross-check is the backstop for paraphrased dodges. This validates AC-1's mechanism before implementation and seeds the implementation's mechanism-first test fixture (the corpus + the labelled edge cases become the guard's test table).
