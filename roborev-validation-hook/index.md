Add a code-quality / subtle-bug review pass to the dev workflow that complements the validation gate's behavioral-parity job. The validation ensign keeps its unique role (behavioral parity against the oracle); the review pass covers code structure, quality, and subtle bugs that parity tests do not catch. Inspired by roborev's review-fix loop. The ideation spike below decided WHERE this pass attaches.

## Problem

The validation gate proves behavioral parity but does not systematically hunt code-quality issues or subtle bugs. The session-1 debrief proposed roborev (an AI-agent code reviewer) as the missing review-quality flywheel. Two candidate operating models were named: (1) ALONGSIDE — per-commit incremental review on the implementation worktree branch, findings feeding the validation gate; (2) PR-MOD — roborev as part of the future pr-merge mod, run at merge time on the whole change. The captain leaned #2. The ideation spike (see Stage Report) probed roborev's real interface and our workflow to decide which model the eventual BUILD targets.

## Spike outcome (the decision)

**Recommended model: #2 (PR-MOD), with a defined fallback.** Grounded in spike evidence:

- roborev exposes TWO architectures. The **daemon architecture** (`install-hook` → `post-commit` → daemon enqueue → `review`/`wait`/`refine`) is roborev's native *incremental/per-commit* path — and it requires a writable `~/.roborev` home + a running daemon. **That home is OS-denied in this sandbox** (`~/.roborev` → `operation not permitted`), and even `review --local` (the documented daemonless single-commit/dirty path) hard-errors on `load config: ~/.roborev/config.toml: operation not permitted`. So the ALONGSIDE model's native mechanism is precisely the sandbox-blocked one.
- The **CI architecture** (`roborev ci review`, configured by repo-local `.roborev.toml`, generated by `roborev init gh-action`) is documented to "run without a daemon or database" and **empirically degrades gracefully when `~/.roborev` is denied** (`load global config: ... operation not permitted (using defaults)` — then proceeds to dispatch the agent). It runs an agent×review-type matrix, synthesizes, and posts a PR comment via `gh pr comment`. This is the path built to run in ephemeral CI where HOME/auth are clean — i.e., the PR-MOD shape.
- Our validation stage already has `feedback-to: implementation`. On REJECTED the FO runs the Feedback Rejection Flow: route findings back into the implementation worktree, reuse the worker handle, re-validate. **This is structurally the same loop as roborev's `refine`** — so we do not need roborev to recreate a review-fix loop we already have; we need roborev to *supply findings into it*.
- FO mods register at lifecycle points **startup / idle / merge** — there is no "per-commit" or "per-implementation-step" lifecycle hook. The pr-merge mod is a **merge-lifecycle** mod, and PR_STATE is already a first-class FO startup concern. PR-MOD therefore attaches at an existing hook point; ALONGSIDE would require inventing a new per-commit hook the FO model does not currently have.
- PR-MOD composes with the two-origin direction: the pr-merge mod already must know which origin/branch each artifact targets (code-repo origin vs the state repo's future orphan-branch origin). `roborev ci review --ref BASE..HEAD` reviews exactly the code-branch range the PR proposes — the code origin, never the `.spacedock-state` checkout — which satisfies the original "watch the CODE branch, not state" intent for free.

**Fallback when no PR/CI host is available** (local-only runs, no GitHub Actions): run `roborev ci review --ref {base}..{head}` directly on the merged code worktree as a **pre-merge review step** invoked by the merge-lifecycle mod, capturing `--comment`-less synthesized output into the entity body / gate. Same command, same daemonless path; the only difference is where the synthesized review lands (entity gate text vs a PR comment).

## Why not ALONGSIDE (#1)

Not chosen, but honestly weighed. Pros: earliest feedback (shift-left during implementation); per-commit granularity matches roborev's `post-commit` design. Cons that decided it: (a) its native mechanism is the daemon path, which is the exact thing the sandbox blocks, and the documented daemonless escape (`review --local`) still hard-errors on the denied home; (b) it would require a new per-commit FO hook point that does not exist in the mods model; (c) it duplicates the review-fix loop the validation `feedback-to` flow already provides; (d) per-commit review on a worktree whose commits are WIP (and will be rebased/squashed at merge under the future pr-mod) reviews throwaway intermediate states. A justified hybrid keeps ALONGSIDE's one durable benefit — findings reaching the validation gate — by having PR-MOD's review run *before* the merge/gate so its synthesized findings are visible at the gate, not after it.

## Acceptance criteria

**AC-1 — The review pass runs at the pr-merge / pre-merge lifecycle point on the code-branch range, never on the `.spacedock-state` checkout.**
Verified by: a pilot pr-merge mod invocation reviews the `BASE..HEAD` range of the entity's code worktree branch (`roborev ci review --ref BASE..HEAD`); the reviewed diff contains only code-repo commits, and no `.spacedock-state` file appears in the reviewed range. Behavioral proof, not a doc assertion.

**AC-2 — Review findings are visible at the decision point (PR comment when a PR host exists; entity gate text otherwise) before the change is accepted.**
Verified by: after the review pass runs, the synthesized findings are retrievable at the gate/PR for the captain's accept/reject decision — when REJECTED, the findings route through the existing validation `feedback-to: implementation` flow exactly like a validation rejection. A pilot shows a seeded code-quality issue surfacing in the gate/PR output and bouncing the entity back to implementation.

**AC-3 — The review pass runs daemonless and survives a denied `~/.roborev`, using claude-code as the review agent.**
Verified by: the pass uses `roborev ci review` (not the daemon/`post-commit` path); with `~/.roborev` denied it logs the `using defaults` fallback and still dispatches the agent; `.roborev.toml` pins `agent = claude-code` (the only agent healthy in this environment — `check-agents` shows codex/gemini/kilo/pi all fail on sandboxed config dirs). When the agent is reachable, a review completes and emits findings; when not (e.g., nested-CLI auth block), the failure is reported, not silently swallowed.

## Test plan

- **Cost/complexity:** the BUILD is a mod, not launcher code — likely low Go surface (the mod is workflow scaffolding + a `.roborev.toml` + a thin merge-lifecycle hook that shells `roborev ci review`). `go test ./...` covers any launcher glue if a native helper is added; otherwise the proof is a live mod pilot.
- **Behavioral pilot (the real proof):** in a code worktree with a seeded code-quality defect, run the merge-lifecycle review step against `BASE..HEAD`; confirm (a) only code-repo commits are in the reviewed range, (b) the seeded defect appears in the synthesized output, (c) a REJECTED outcome routes back through `feedback-to: implementation`. Live workflow test — runtime behavior is the claim.
- **Sandbox/agent proof:** assert `roborev ci review` runs daemonless with `~/.roborev` denied (the `using defaults` path), and that `.roborev.toml` pins claude-code. The agent-auth reachability is environment-dependent (see Notes) and is reported, not asserted green, in this sandbox.

## Notes

roborev v0.56 is installed. **Sandbox reality (spiked):** the daemon path is hard-blocked (`~/.roborev` OS-denied). `HOME` relocation gets roborev *past* its config-load and lets it invoke an agent, but it does NOT cleanly fix the build here: claude-code's own auth is keyed to the macOS login keychain (`Claude Code-credentials`, not a file under `~/.claude`) plus `$HOME/.claude.json` session state — so a relocated `HOME` leaves claude-code reporting "Not logged in." `check-agents` confirms claude-code is the only healthy agent (codex/gemini/kilo/pi all fail on denied config dirs). Running roborev's claude-code review from *inside* a claude-code agent session is additionally a nested-CLI auth case. **This is why the BUILD stays deferred:** roborev's review agent cannot authenticate in this sandboxed nested context. The BUILD un-defers when EITHER (a) the pr-merge mod exists AND roborev runs in a clean CI host (GitHub Actions, where `roborev init gh-action` + repo secrets give the agent a real key — the designed-for case), OR (b) a local run can give the review agent working auth (non-nested HOME with valid keychain/credentials). PR-MOD itself is post-bootstrap (README: "No PR merge flow, mods, or lifecycle hooks are in scope for this bootstrap workflow"), so this BUILD is gated behind the pr-merge mod landing first. Lower priority than the self-hosting entities; genuine review-quality flywheel, not on the critical path.

### Summary

This was a forensic operating-model spike, not a build. roborev exposes two architectures; the per-commit/ALONGSIDE one is wedded to the daemon home that this sandbox denies (even `review --local` hard-errors on `~/.roborev/config.toml`), while the CI/PR-shaped `ci review` runs daemonless and degrades gracefully — confirming the captain's lean toward model #2 (PR-MOD) on sharper evidence than "it ties into pr-mod." The recommendation is PR-MOD with a no-PR-host fallback, attaching at the existing merge lifecycle point and feeding findings through the validation gate's existing `feedback-to: implementation` loop rather than duplicating it. The decisive unknown-unknown the spike surfaced: HOME relocation does NOT cleanly unblock the build, because claude-code's auth is keyed to the macOS keychain + `$HOME/.claude.json` session state, so a relocated HOME breaks the review agent's auth — which is the concrete reason the BUILD stays deferred until a clean-auth CI host (or non-nested local auth) plus the post-bootstrap pr-merge mod both exist.
