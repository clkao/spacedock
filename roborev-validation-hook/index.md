Add an AI code-quality / subtle-bug review pass to the dev workflow that complements the validation gate's behavioral checks. The validation ensign keeps its unique role (behavioral parity against the oracle, AC-by-AC evidence reproduction); the review pass covers code structure, quality, and subtle bugs that AC checks do not systematically hunt. The pass runs roborev's **multi-agent fanout + synthesis** — ≥2 independent review agents over the diff, synthesized into one combined result — not a single-agent pass. Inspired by roborev's review-fix loop. This re-ideation decides WHERE the pass attaches under the captain's relaxed-sandbox direction and pins the fanout as a first-class requirement.

## Problem

The validation gate proves the deliverable meets its acceptance criteria, but it does not systematically hunt code-quality issues or subtle bugs across the diff. The session-1 debrief proposed roborev (an AI-agent code reviewer) as the missing review-quality flywheel. Two candidate operating models were named: (a) VALIDATION-PASS — a review pass the FO runs as part of the validation stage on the implementation worktree branch, findings feeding the existing validation `feedback-to: implementation` loop; (b) PR-JOB — a label-gated PR review job riding the live-e2e CI footing, run on the PR diff. The point is **coherent development ergonomics**: a review-quality pass that feels like a first-class, natural part of the dev flow — not a someday-deferred hook.

## Reframe: the sandbox constraint that deferred this is RELAXED (captain, 2026-05-31)

The prior ideation cycle deferred the BUILD because `~/.roborev` is OS-denied in the sandbox and roborev's nested claude-code review agent "cannot authenticate in this sandboxed nested context." The captain has relaxed that: roborev runs **unsandboxed with real claude-code auth**, exactly as the just-archived `live-e2e-ci-footing` (id b6tef0q53k5v9d3vsga49sz4) proved — nested claude-code + macOS-keychain credential + sandbox-relaxed execution all work, and the FO ran that live cycle locally itself. The `~/.roborev` denial and the nested-auth block were sandbox-EXPLORATION artifacts, not real blockers.

**This re-ideation re-ran the riskiest path under the relaxed constraint and it WORKS — even from inside this sandboxed ensign.** `roborev ci review --ref HEAD~1..HEAD --agent claude-code` on a seeded SQL-injection diff (`/tmp/rrtest`): roborev logged `load global config: ~/.roborev/config.toml: operation not permitted (using defaults)`, ran daemonless, dispatched the nested claude-code agent, and emitted a real finding — `Severity: critical, Location: vuln.go:12-13, SQL injection via string concatenation` with exploit path and a parameterized-query remediation. The agent authenticated and reviewed; only codex/gemini/kilo/pi fail (their config dirs are denied). So the prior "review agent cannot auth nested" conclusion does NOT hold for claude-code, the agent we pin. The BUILD is unblocked NOW. The prior "deferred until a clean CI host" framing is superseded.

## Recommended model: VALIDATION-PASS (model a), with PR-JOB as the CI extension

**Attach the review pass to the existing validation stage**, not a future pr-merge mod. Rationale grounded in the relaxed-constraint evidence and the actual workflow shape:

- **Coherent ergonomics without inventing a lifecycle.** The README is explicit: "No PR merge flow, mods, or lifecycle hooks are in scope for this bootstrap workflow." PR-MOD (the prior recommendation) is gated behind a mod system that does not exist. The **validation stage exists today** — it is `worktree: true`, `fresh: true`, `gate: true`, `feedback-to: implementation`. A roborev pass run during validation rides all of that for free: it lands at the existing accept/reject gate, and on REJECTED its findings route through the existing `feedback-to: implementation` flow exactly like any validation rejection. This is the natural, first-class dev-flow integration the captain asked for — no new hook point.
- **The review-fix loop already exists; roborev only supplies findings into it.** On REJECTED the FO runs the Feedback Rejection Flow: route findings into the implementation worktree, reuse the worker handle, re-validate. This is structurally roborev's `refine` loop. We do not rebuild it; the validation review pass feeds it.
- **Complements, does not duplicate, the validation ensign.** The validation ensign reproduces AC evidence and judges behavioral correctness. roborev hunts cross-cutting code-quality/security defects on the whole diff via a **multi-agent fanout** (the security review type surfaced the seeded SQLi above). They are orthogonal review lenses at the same gate — this session's adversarial-audit pattern (the FO re-auditing security surfaces before merge) is the human analog roborev automates, and the multi-agent fanout multiplies it: independent models (claude-code, codex, …) each review the diff and roborev synthesizes their findings, so a defect one model misses another can catch.
- **Range scoping is structural under split-root.** The code worktree is a SEPARATE git repo from `.spacedock-state`. `roborev ci review --ref BASE..HEAD` on the code worktree branch reviews exactly the code commits and physically cannot see `.spacedock-state` (it is not in that repo). The "watch the CODE branch, not state" intent holds for free — confirmed: `roborev ci review --ref` scopes identically to `git diff <range>`.
- **The FO runs it the same way it ran the live-e2e cycle.** The decisive proof that this is run-NOW, not deferred: the FO already runs nested claude-code unsandboxed locally (it ran the gated live-e2e cycle that way). `roborev ci review` is the same unsandboxed-nested-claude shape. The actual roborev pass during a real validation is **FO-RUN** (the nested claude-code auth needs the unsandboxed + keychain path, exactly like the live-e2e spike) — flagged for the FO, not the sandboxed ensign.

**PR-JOB (model b) is the CI extension, not a competing model.** It rides the just-shipped live-e2e footing: `runtime-live-e2e.yml` already proves `pull_request: branches: [next]` + a `run-live-e2e` label gate + the CI-E2E* required-reviewer environments + agents installed and fed their per-runtime secret. A roborev PR job is the SAME shape — `roborev init gh-action --agent claude-code,codex` generates exactly that workflow (`pull_request` trigger, `roborev ci review --ref base..head --comment --pr N`, BOTH `ANTHROPIC_API_KEY` + `OPENAI_API_KEY` secrets, BOTH agent CLIs installed). So PR-JOB is a near-free second surface once the local VALIDATION-PASS is proven: the same multi-agent `roborev ci review` command, the same fanout agents, the same proven CI gate; the only delta is `--comment` posting the synthesized result to the PR instead of the FO folding it into the gate. Recommendation: BUILD the VALIDATION-PASS first (it is the coherent local dev-flow win and needs no CI), and offer the PR-JOB as a thin follow-on that reuses the live-e2e CI footing. See "Multi-agent fanout + synthesis" below for the agent×environment mapping.

## Why not PR-JOB-only (the prior recommendation, superseded)

The prior cycle picked PR-MOD because (a) the daemon path was sandbox-blocked, forcing the CI-shaped command, and (b) the FO mod model lacked a per-commit hook. Both premises are now weaker: (a) `roborev ci review` is the daemonless command we use in BOTH models — it is not a PR-only path, it runs fine locally during validation (proven above); (b) we are NOT proposing a per-commit hook — we attach at the EXISTING validation gate, which the FO already drives. PR-JOB-only would defer the entire benefit behind GitHub Actions plumbing and a PR flow the bootstrap workflow does not yet run, when the same review can run locally at the validation gate the FO already operates today. PR-JOB stays as the valuable CI extension, not the primary model.

## Multi-agent fanout + synthesis (first-class requirement)

The review pass MUST run roborev's native agent×review-type matrix with **≥2 review agents** and let roborev **synthesize** their findings into one combined result — the synthesis is what lands at the gate (VALIDATION-PASS) or as the PR comment (PR-JOB). It must NOT collapse to a single claude-code pass; the prior cycle pinned one agent only because the sandbox left claude-code as the only healthy one, and that constraint is relaxed. roborev's `ci review` is documented as exactly this: "Run the full review_type x agent matrix in parallel, synthesize results, and output or post a PR comment" (flags `--agent`, `--review-types`, `--synthesis-agent`).

**Fanout agents + authentication.** Minimum fanout = **claude-code + codex**; add **gemini** (and others) only if they authenticate unsandboxed (the FO confirms via `roborev check-agents` on its real machine — in the sandbox gemini/codex/pi all fail on denied config dirs, but those denials are sandbox artifacts).

| Agent | Credential (FO-run VALIDATION-PASS) | Credential (PR-JOB) | CI-E2E* environment |
|---|---|---|---|
| claude-code | macOS login keychain (`Claude Code-credentials`) — the live-e2e auth path | `ANTHROPIC_API_KEY` secret | CI-E2E |
| codex | local `~/.codex` auth (real machine) | `OPENAI_API_KEY` secret | CI-E2E-CODEX |
| claude-code (opus) | same keychain, opus model | `ANTHROPIC_API_KEY` secret | CI-E2E-OPUS |

Both `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` already exist on `spacedock-dev/spacedock`, and the three approval-gated environments (CI-E2E / CI-E2E-CODEX / CI-E2E-OPUS) the live-e2e work proved map 1:1 onto these agents.

**`.roborev.toml` config (the committed spec).** Pin the agent list + review-type matrix + synthesis agent under `[ci]` so every run fans out consistently:

```toml
[ci]
agents = ["claude-code", "codex"]        # ≥2; FO adds gemini if it auths unsandboxed
review_types = ["security", "quality"]   # the agent×type matrix roborev fans out
synthesis_agent = "claude-code"          # the agent proven to auth nested; combines per-agent findings
reasoning = "thorough"
```

`roborev init gh-action --agent claude-code,codex` generates the PR-JOB workflow that installs BOTH agent CLIs and wires BOTH secrets into one `ci review` step (verified — see Stage Report). The `--agent`/`--review-types`/`--synthesis-agent` CLI flags override the `.toml` for the FO's local run.

**Synthesis behavior + agent-failure handling (spike-observed).** When one agent fails to auth, roborev does NOT fail the whole pass: it emits `> Synthesis unavailable. Showing individual review outputs.` and a `## roborev: Combined Review` listing each agent's result labeled `(done)` / `(failed)`. So a per-agent failure is surfaced, not swallowed — the FO sees which legs ran. When all fanout legs succeed, roborev's `synthesis_agent` combines them into one synthesized result; THAT synthesized result (not the raw per-agent dumps) is what the FO records at the gate.

**PR-JOB environment mapping.** Each fanout agent's leg rides its own approval-gated environment exactly as `runtime-live-e2e.yml`'s matrix proved (sonnet→CI-E2E, opus→CI-E2E-OPUS): the roborev PR job runs `ci review --agent claude-code,codex` with both secrets present, and roborev fans out + synthesizes across the legs in one job. (If the captain prefers per-agent approval isolation, the legs can be split into a matrix of jobs — one environment each — that each emit a partial review and a final synthesis job combines them; the single-job multi-secret form is simpler and is what `init gh-action` generates.)

## Acceptance criteria

Each AC names a property of the finished entity, not a stage action, and how it is verified.

**AC-1 — The review pass reviews exactly the code worktree branch's `BASE..HEAD` range and never the `.spacedock-state` checkout.**
Verified by: during a validation pilot, the FO runs `roborev ci review --ref BASE..HEAD` on the entity's code worktree branch; the reviewed range (asserted via `git diff --name-only BASE..HEAD` equivalence) contains only code-repo files, and because the code worktree is a separate git repo from `.spacedock-state`, no state file can appear. Behavioral proof on a real range, not a doc assertion.

**AC-2 — A seeded code-quality/security defect surfaces in the synthesized review output at the validation gate and bounces the entity back to implementation.**
Verified by: in a code worktree carrying a seeded defect (e.g. the SQL-injection fixture this spike used, or an equivalent), the validation review pass emits a synthesized finding naming the defect (severity + file:line), the FO records the SYNTHESIZED result (not raw per-agent dumps) in the entity gate text, and a REJECTED outcome routes the finding through the existing validation `feedback-to: implementation` flow — the entity returns to `implementation` with the finding as feedback, exactly like a validation rejection. Live workflow pilot — runtime behavior is the claim.

**AC-3 — The review pass runs daemonless and reports agent failure rather than passing silently.**
Verified by: the pass uses `roborev ci review` (not the daemon/`post-commit` path); it runs with `~/.roborev` absent/denied, logging `using defaults`. With working auth (the FO's unsandboxed local run via keychain/`~/.codex`, or the per-runtime secret in CI) the agents complete reviews and emit findings (spike-proven: the seeded SQLi surfaced from claude-code); when an agent is unreachable, roborev surfaces it as a labeled `(failed)` leg in the combined output and the FO sees it at the gate — not swallowed (spike-observed: the sandbox-denied codex leg reported `(failed)` with its error rather than silently dropping).

**AC-4 — The review pass fans out across ≥2 review agents and roborev synthesizes their findings into one combined result that is what lands at the gate / PR comment.**
Verified by: `.roborev.toml` (or the `--agent` override) lists ≥2 agents (claude-code + codex minimum); a pilot run logs `agents=[claude-code codex …]` and a parallel `running agent=… ` line per agent (spike-proven: both legs dispatched in parallel), and roborev runs a `synthesis` step combining them; the FO records the single synthesized `## roborev:` result at the gate, not the per-agent dumps. The fanout is real, not nominal: the pilot shows ≥2 distinct agents actually reviewing the same diff and the synthesis agent merging them. FO-RUN (codex needs unsandboxed `~/.codex` auth — see Test plan).
